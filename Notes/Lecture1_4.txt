REST (Representational State Transfer) is a style of software
architecture for distributed hypermedia systems such as
the World Wide Web.
’ü« Proposed by Roy Thomas Fielding in his doctoral dissertation
"Architectural Styles and the Design of Network-based
Software Architectures" in 2000.
’ü« Fielding is one of the principal authors of the HTTP
specification versions 1.0 (1996) and 1.1 (1999).
’ü« Services developed based on REST concept and conforming

http://en.wikipedia.org/wiki/Representational_State_Transfer

REST Concept

8

REST concept is about how communication is done between clients
and servers using HTTP at the application level. REST concept is
defined by these principles:
A client initiates a request to a server;
’ü« The server processes the request and returns a response:
’ü« The response can be a static data
’ü« The return value of an operation
’ü« Communication is stateless;
’ü« Communication is about the "representations" transfer of
"resources" between a client and a server;
’ü« Each resource (or a set of resources) is given a unique identifier,
called URI (Universal Resource Identifier).
’ü«

Y. Chen

REST Architecture and REST Constraints

9

REST architecture is a software development style that
follows these design constraints and guidelines:
Client-server separation: Clients and servers are
logically separated and communicate through a uniform
interface conforming to HTTP.
’ü« Stateless: Every request from a client is considered
from a new client by the server.
’ü« Client-side cacheable: The response delivered from the
sever must contain the complete information for the
client to reuse the response, if the client requests the
same URI. ŌĆō Output Caching on client side (Lecture 3-4
and Textbook 5.5).
Y. Chen
’ü«

REST Architecture and REST Constraints (contd.)

10

Layered architecture: A server is a logical unit with a
layered architecture.
’ü« A client identifies a server by an URI.
’ü« The resource may or may not reside on the server
that the client is communicating with.
’ü« The request can be redirected to another server
transparently.
’ü« Multiple redirections are possible.
’ü« Code on demand: A server may process the request by
executing a program on the server side or sending the
code (script) to the client side to process.
’ü«

Why Microservices?
Microservices and microservice application architecture
are a software development technique, a variant of SOA style. It
helps organizing large number of services in a more organized way.
’ü« If an application comprises of a dozen services, no worries about
how these services connect to each other. We can use the
existing software architectures: blackboard, pipeline, bus, pointto-point, etc.
’ü« If an application comprises of a hundred or thousand (typically,
smaller or micro) services, what should we do?
’ü« Using the current architecture is possible, but it may become
too complex.
’ü« New architectures that are more regular, with certain
restrictions to the flexibility of connections, e.g., connect to
neighboring nodes only.
Y. Chen
